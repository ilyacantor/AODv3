## C. Rules of Behavior

### C1) Determine merge_key

For each asset:

* compute `registered_domain` (eTLD+1) from the best available domain candidate:

  * prefer `asset.identifiers.domains` if present
  * else if `asset.name` looks like a domain
  * else none

If `merge_key` is empty → **do not merge** (asset stays standalone; no synthetic key). This preserves determinism and avoids name-based accidental merges.

### C2) Fan-in merge grouping

Group assets by `merge_key` where `merge_key` is non-empty.

### C3) Deterministic winner selection

Winner = asset with lexicographically smallest `asset_id` (or next best stable deterministic identifier if `asset_id` isn’t directly comparable).

No timestamps for “oldest”. Keep it deterministic across runs.

### C4) Field-by-field merge (must implement exactly)

Implement the merge table:

* `asset_id`: **winner keeps**
* `name`: **set to canonical registered domain** (`merge_key`) for merged groups
* `identifiers.domains`: **union + ordered** (see ordering below)
* `identifiers.hostnames`: **union** (dedupe)
* all other identifier lists (ids, urls, etc. if present): **union** (dedupe)
* `evidence_refs`: **union** (dedupe) — sort for deterministic diffs
* coverage booleans (`has_idp`, `has_cmdb`, `has_cloud`, `has_finance` or equivalents): **OR**
* `activity` (latest activity timestamp / evidence): **max()**
* `lens_statuses` / derived classifications: **recompute** from merged flags + merged activity using the existing policy semantics (governed=idp OR cmdb; zombie=governed AND stale AND ongoing_finance; shadow=ungoverned AND recent). 

Fields to **NOT** merge (winner keeps; log conflict if mismatched):

* `asset_type`
* `environment`
* `provisioning_status`
* `lens_debug` / `match_info` / debug payloads (keep winner’s, optionally attach a lightweight `merge_note` if such a field exists)

### C5) Domain ordering rules (canonical at index 0)

After merge:

1. `domains[0] = merge_key` (canonical)
2. then winner’s prior domains in their original order (excluding canonical)
3. then the remaining domains from all other assets, lexicographically sorted (excluding already included)

This preserves “winner priority” while staying deterministic.

## D. Reconciliation Adapter Hardening (Required)

Update the reconciliation lookup so Farm can match by:

1. `asset.name == requested_domain`
2. OR `requested_domain in asset.identifiers.domains`

This prevents future key drift if naming changes again.

(Reason: onboarding docs explicitly state Farm expects domain keys and this mismatch is the core issue.)

## E. Acceptance Criteria (Hard Gates)

Run:

* `POST /api/discovery/run`
* `GET /api/reconciliation/assess/{run_id}` 

Must pass all:

1. **Shadows**

* Matched ≥ 56 and FP = 0 (do not break current shadow performance) 

2. **Zombies**

* Matched = 56 (or within 1 with explicit RCA)
* FP ≤ 1 

3. **Rejections**

* Must not spike relative to baseline (no return of “Domain Guillotine” behavior; prior bad run rejected 550/1163) 

4. **Determinism**

* Re-running same snapshot produces identical asset keys and counts (no nondeterministic ordering).

If any gate fails → rollback.
