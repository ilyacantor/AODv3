## Required Checks (Phase 0)

### 1) Storage modality audit (dual-store detection)

Detect if AOD is persisting the same “assets/snapshots” in multiple places, e.g.:

* SQLite file(s)
* Postgres/Supabase (via env var)
* JSON dump folders
* Cache layers that masquerade as truth

**Rules**

* If more than one persistent store is present AND non-empty for the same logical data class → **FAIL**
* If a legacy directory exists with data (json/csv/parquet) that looks like assets/snapshots → **FAIL**
* If store is present but empty, print as INFO.

Implementation guidance:

* Search repo for typical folders: `data/`, `snapshots/`, `exports/`, `tmp/`, `.cache/`
* Identify DB file candidates (`*.db`, `*.sqlite`, `*.sqlite3`)
* Read-only stat checks + file counts

### 2) “Deterministic-only” policy scan (static)

We want to catch drift back into:

* anomaly scoring
* ML heuristics
* training center knobs affecting zombie classification
* “contamination”, “threshold”, “score”, “probability”

**Rule**

* This is Phase 0: emit **WARNINGS** (not FAIL yet), but print the exact file paths + line numbers.

Implementation:

* Grep-like scan (python stdlib) for keywords:

  * `anomaly`, `contamination`, `isolationforest`, `threshold`, `score`, `probab`, `confidence`, `heuristic`, `ml`
* Print findings grouped by file.

### 3) Zombie classification contract presence check (static)

We need to ensure AOD has an explicit, deterministic rule path for zombies (or at least no “placeholder ambiguity”).

**Rule**

* If there is no clear reference to `observed_at` / `last_seen` / “activity evidence” fields anywhere in code/config → **FAIL**
  (Because then “zombies” can only be guessed.)

Implementation:

* Search for required evidence keys:

  * `observed_at`, `last_seen`, `last_activity`, `activity`, `evidence`
* If none found, fail with message: “No activity evidence fields found; zombie logic cannot be deterministic.”

### 4) Timestamp integrity guardrail (behavioral, read-only)

If AOD has a local DB file, inspect it directly:

* List tables
* Print columns for asset/activity/evidence tables (best effort)
* Flag if any timestamp columns are being defaulted via DB default to `CURRENT_TIMESTAMP` (warn for now)

If AOD does not have a local DB, inspect environment:

* Print whether `DATABASE_URL` / `SUPABASE_*` / `REPLIT_DB_URL` exist (just presence, no secrets)

**Rule**

* If AOD is injecting timestamps on ingest when upstream doesn’t provide them, we must detect it later — so for Phase 0: print warnings if defaults exist.

### 5) API smoke (optional, secondary)

If a local server is running on a known port (e.g., 5000), do a quick GET to:

* `/health` or `/docs` or `/dashboard/assets?page_size=1`
  But:
* Do **not** start the server
* Do **not** rely on this for PASS
* Only print INFO/WARN if unreachable
