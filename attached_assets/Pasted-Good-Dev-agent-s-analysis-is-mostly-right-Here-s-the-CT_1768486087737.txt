## 1) CMDB `record.domain` → `identifiers.domains`

**Approve. Implement now.**

But do it with guardrails:

**Validation gate for CMDB domains**

* must parse as a domain (no URL, no path, no scheme)
* must not be an IP
* must not be in shared infra / vendor root / generic “generic collision roots” *unless* the asset is CMDB-anchored and CI type supports it
* optional: require presence of a dot + reasonable TLD length

**Why this is safe**

* It doesn’t undo Stage 1 fix (external_ref URL leakage) because you’re using `record.domain`, not `external_ref`.
* It will reduce the “name-only vs domain-key” split immediately.

## 2) Domain provenance: do NOT blow up the schema

Dev is right: changing `domains: list[str]` to objects will cascade into DCL/Farm/UI.

**CTO decision: add a parallel field, keep `domains` as-is.**

Add:

* `identifiers.domain_provenance: dict[str, str]` (domain → source enum)
* `identifiers.domain_confidence: dict[str, float]` (optional, later)
* keep `identifiers.domains: list[str]` for backward compat

Where `source ∈ {discovery, cmdb, idp, vendor_map, inferred}`

This cleanly separates:

* **match_method** (how correlated)
* **domain_source** (where a domain came from)

No conflation.

## 3) Key generation: align, but VERSION IT

Dev is also right: changing `compute_canonical_key()` without versioning will wreck historical reconciliation.

**CTO decision: implement v2 key strategy but keep v1 available.**

### Key strategy versioning

* Add `policy_snapshot.key_strategy_version = "v1" | "v2"`
* Store it in run metadata and in every `ReconciliationResult`
* During reconciliation:

  * If Farm and AOD versions mismatch, compute **both keys** and compare using a mapping.

### v2 algorithm (what I want)

`entity_key_v2` priority:

1. **discovery registered_domain** (domain_source=discovery)
2. **cmdb_primary_domain** (domain_source=cmdb, only if CMDB matched)
3. **idp_app_id / service_principal_id** (only if IdP matched and stable ID exists)
4. **name_slug** fallback (stable normalization)

**Important nuance**: dev flagged a real gap:

> current code can’t prioritize discovery domain vs CMDB domain because both land in `domains[]`

Fix that by using `domain_provenance` to pick the highest-priority domain, not `domains[0]`.

### What to do about “idp_app_id as key”

Implement it in v2 **only if you can guarantee stability**:

* Prefer a stable IdP object ID (app object id / service principal id), not display name.
* If you don’t have that reliably today, skip step (3) for now and log “IDP_KEY_MISSING”.
