## 1) Late-binding: exact code path + what changes

### Where it goes

In `execute_pipeline()` **after** the Admission loop finishes building `assets[]`, and **before** `generate_findings()` runs.

You can see the pipeline builds `assets` inside the admission loop, then immediately calls `generate_findings(assets, ...)`. 

So we insert one step:

```
assets = late_bind_and_merge_assets(assets, correlations)
findings = generate_findings(assets, correlations, ...)
```

### What it modifies

* **Modify Asset objects in place** (preferred, minimal churn).
* Only touch:

  * `asset.name`
  * `asset.identifiers.domains` (ensure canonical + all variants)
  * optional: `asset.identifiers.hostnames` (preserve original_name, if not already)

We do **not** mutate:

* `candidate.entity_id`
* correlation structures
* rejections / ambiguous matches rows

This aligns with the “identity fixed early, naming decided late” invariant already documented. 

### How we pick canonical domain

Do **not** invent a new extractor.

Use admission’s existing domain harvesting intent (it already contains correlation-domain extraction utilities and the “include ALL domains” goal). 

Rule:

1. `canonical = extract_registered_domain(best_domain_candidate)`
2. Best candidate priority:

   * domains found in correlated plane records (IdP/CMDB first)
   * then candidate.domain
   * then leave name alone

(You already have `_extract_domain_from_correlation()` and `_extract_all_domains_from_correlation()` in admission, which is exactly this purpose. )

**Impact:** naming + reconciliation improve without destabilizing correlation/admission.

## 2) Fan-in merge: deterministic algorithm + conflict rules

### Deterministic key

`merge_key = registered_domain` (eTLD+1).

### Deterministic “oldest”

Not timestamps (often missing / inconsistent). We use **stable ordering**:

> Winner = asset with lexicographically smallest `asset.asset_id` (or if absent, smallest `entity_id` inside identifiers).

This is deterministic across runs because IDs are deterministic UUIDs in this codebase pattern. 

### Merge semantics (no surprises)

When N assets share merge_key:

**Union / OR fields**

* `identifiers.domains` = union (dedupe)
* `evidence_refs` = union (dedupe)
* `lens statuses / has_idp / has_cmdb / has_finance` = OR
* `has_ongoing_finance` = OR

**Activity**

* `latest_activity_at` = **max** of the group.

  * If *any* evidence is recent, the merged thing is not stale. That is correct.

**Classification**

* Recompute **derived classification** on the merged asset from the merged flags + activity.

  * Don’t “pick one asset’s classification”.
  * Don’t average.
  * Use the same rules you already document: governed = idp OR cmdb; zombie = governed AND stale AND ongoing_finance; shadow = ungoverned AND recent. 

**Deduping**

* `evidence_refs`: set() then stable sort before storing to keep deterministic diffs.

**Impact:** collisions stop causing UUID/PK chaos, and the resulting classification is logically consistent.

## 4) Child records: what happens on merge

Key point: **we are not merging entities pre-admission**. We’re merging **Assets post-admission**.

So:

* `ambiguous_matches` stays exactly as recorded per entity (and is already deduped by match_id) 
* `rejections` stay exactly as recorded per entity (deduped by rejection_id) 
* `obs_samples` unchanged (deduped already) 

What we must handle:

* **Findings** generated per-asset: if we merge assets *before* `generate_findings()`, this becomes a non-issue because findings get generated once for the merged asset. That’s why the insertion point matters. 

**Impact:** we avoid orphaned findings pointing at “asset_ids that disappeared”.
