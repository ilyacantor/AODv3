## Stage 2: “Real Farm + Lifecycle + Findings That Actually Match”

**Goal of Stage 2:**
Turn AOD v3 from “toy ingest + counts” into a **credible mini-product**:

* Use the **real Farm enterprise snapshot** shape (surfaces, signals, farm_asset_id)
* Persist **one clean asset row per `farm_asset_id`** with lens coverage and core ontology.
* Apply a **single lifecycle router** with the correct blocking rules.
* Generate **Findings** from CMDB signals (shadow_it, governance_gap, data_conflicts, ops_risk, low_confidence).
* Make the **dashboard, triage cards, and asset detail drilldowns all agree numerically**.

No ML / Training Center / experiments. Just rules + Farm.

Concretely, Stage 2 should give you:

1. **Farm ingest uses real contract**

   From Farm docs: AOD should consume `POST /api/farm/enterprise/new` with `archetype` + `scale`, and read CMDB conflict signals from `surfaces.cmdb.evidence[*].signals`.

   Key signals you must wire:

   * `signals.farm_asset_id` → `assets.farm_asset_id` (stable identity)
   * `signals.parked_reason`
   * `signals.has_data_conflicts`
   * `signals.conflict_types[]`
   * `signals.rules_triggered[]`
   * `signals.is_shadow_it`
   * `signals.anomaly_score`
   * `signals.prob_kind`

2. **Lens coverage & shadow IT basics**

   * Build `lens_coverage` per asset, keyed by lens name (`cmdb`, `idp`, `saas_api`, `billing`, `edr`, `browser`, `network`, `observability`).
   * For now:

     * **Primary truth** for `is_shadow_it` = Farm’s `signals.is_shadow_it` (if present).
     * Later you can add lens-pattern heuristics, but Stage 2 can just mirror Farm.

3. **Canonical lifecycle routing (one place only)**

   Implement **one** router that decides `lifecycle_state` + `parked_reason` from CMDB signals:

   * Blocking → `PARKED`:

     * Rule IDs: `SOR_CONFLICT`, `ONT_SOR_CONFLICT`, `SCHEMA_MISMATCH`, `SCHEMA_OR_SHAPE_MISMATCH`, `DATA_SCHEMA_DRIFT`, `ID_COLLISION`, `MISSING_PRIMARY_ID`.
   * Everything else → `CATALOGED`.
   * No one else in the codebase is allowed to mutate lifecycle_state (except a future explicit HITL endpoint).

4. **Findings generation from signals**

   For each asset, based on CMDB signals + derived fields, generate rows in `findings`:

   * `shadow_it`

     * Condition: `is_shadow_it = true`.
     * Severity: `warn` unless conflict types escalate it.
   * `governance_gap`

     * Condition: all of `owner`, `owner_email`, `owner_team` are null/empty **or** the vendor is unmapped.
   * `data_conflicts`

     * Condition: `has_data_conflicts = true` or `conflict_types` non-empty.
   * `ops_risk`

     * Condition: `anomaly_score >= 0.4`. 
   * `low_confidence`

     * Condition: `prob_kind < 0.5`.

   All of these **must** leave lifecycle_state alone. They’re **reports**, not blockers.

5. **Dashboard & triage that actually reconcile**

   Update the dashboard + triage queries so that:

   * Lifecycle cards:

     * `PARKED` = count of assets with `lifecycle_state='PARKED'`.
     * `CATALOGED` = ditto.
   * Blocking triage cards:

     * SoR Conflict: `PARKED` with `parked_reason` including `SOR_CONFLICT` / `ONT_SOR_CONFLICT`.
     * Schema Mismatch: `parked_reason` includes `SCHEMA_MISMATCH` / `SCHEMA_OR_SHAPE_MISMATCH` / `DATA_SCHEMA_DRIFT`.
     * ID Collision: `parked_reason='ID_COLLISION'`.
     * Missing ID: `parked_reason='MISSING_PRIMARY_ID'`.
   * Non-blocking triage cards:

     * Shadow IT: count of `findings.finding_type='shadow_it'`.
     * Governance Gap: `finding_type='governance_gap'`.
     * Data Conflicts: `finding_type='data_conflicts'`.
     * Ops Risk: `finding_type='ops_risk'`.
     * Low Confidence: `finding_type='low_confidence'`.

   Clicking any card → asset list → asset detail, with numbers that line up 1:1 with those queries.
