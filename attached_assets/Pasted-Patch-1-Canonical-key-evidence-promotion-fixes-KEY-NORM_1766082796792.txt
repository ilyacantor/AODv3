Patch 1 — Canonical key + evidence promotion (fixes KEY_NORMALIZATION_MISMATCH)
Goal

If AOD has any evidence that contains a domain/hostname, it must deterministically:

extract hostname

normalize it

derive canonical asset key (usually eTLD+1)

promote that key into the admitted asset set (even if later filtered)

Implementation checklist

A. Normalize every hostname the same way (single function, used everywhere)

lowercase

strip scheme (https://)

strip path/query/fragment

strip port (:443)

strip trailing dot

decode punycode → unicode or keep punycode but be consistent

handle IPs (return None or treat as separate asset type)

B. Derive canonical domain deterministically

Use a Public Suffix List library to get eTLD+1 (e.g., foo.bar.co.uk → bar.co.uk)

If hostname is already eTLD+1, keep it

If hostname is a known “service domain” (e.g., *.googleusercontent.net), you still promote googleusercontent.net as key (don’t drop because it’s “weird”)

C. Promote canonical keys from all planes (this is the usual bug)

discovery observations (dns/proxy/browser/endpoint/network_scan/etc.)

idp objects (app URLs, issuer URLs, domains)

cloud resources (endpoints, domains)

finance records (contract domains, vendor URLs, invoice URLs)

D. Make promotion explicit
Have a stage like extract_candidate_asset_keys(snapshot) -> Set[str]
and log counts by source:

candidate_keys_from_discovery

candidate_keys_from_idp

candidate_keys_from_finance

candidate_keys_from_cloud

Acceptance test (must pass)
For each “missed_by_aod + KEY_NORMALIZATION_MISMATCH” item, assert:

the domain exists in candidate set

the domain becomes an admitted asset (before classification)

classification runs (even if later suppressed)

This directly targets the “AOD has evidence … but did not normalize to domain key” misses

Patch 2 — Finance→Asset linking and reason-code truthfulness
Goal

If a finance record exists for a domain/vendor, HAS_FINANCE should be true for the corresponding canonical asset key—and never silently lost.

Implementation checklist

A. Normalize finance “targets” into one of:

domain (best)

vendor name (fallback)

product name (fallback)

B. Deterministic linking priority

If contract/invoice has a URL/domain → normalize + eTLD+1 → link to that asset key

Else if it has a vendor domain in metadata → link

Else link by vendor slug/name to an internal vendor entity (not domain), and mark reason codes:

HAS_FINANCE_VENDOR_ONLY (truthful) vs HAS_FINANCE_DOMAIN_LINKED

C. Keep recurring vs one-time deterministic

set HAS_ONGOING_FINANCE only when recurring/contract/subscription conditions are met

ensure you don’t downgrade to NO_FINANCE just because it’s “vendor-only”

Acceptance test
Pick examples where Farm expects finance and make sure AOD reason codes reflect it (even if asset remains shadow for other reasons). The current mismatch is visible (e.g., hubspot.com shows Farm finance but AOD reason codes include NO_FINANCE)

Patch 3 — Zombie policy determinism (reason codes and windows)
Goal

Zombie/shadow decisions are deterministic, explainable, and consistent across runs.

Implementation checklist

A. One function owns “activity recency”

RECENT_ACTIVITY vs STALE_ACTIVITY computed from the best timestamp available

deterministic “best timestamp wins” ordering (e.g., IdP last_login > SaaS audit log > proxy/browser > finance last_payment)

B. Reason codes must match the evidence
If you say NO_FINANCE, it must mean “no linked finance record exists”, not “finance exists but I couldn’t join it”.

C. Add a reconciliation-mode diagnostic
For any zombie asset:

print which evidence timestamp drove recency classification

print which plane contributed which flags

Acceptance test
For a small set of known zombies, the same input snapshot yields identical:

zombie/shadow classification

reason code set

activity timestamp used

Order of operations (do this in sequence)

Patch 1 (key normalization + promotion) — this will collapse the giant “missed_by_aod” class.

Patch 2 (finance linking + truthful reason codes) — fixes correctness of finance flags and downstream policies.

Patch 3 (zombie determinism + explainability) — cleans drift and makes future debugging sane.
