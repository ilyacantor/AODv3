### Review of Issue 1: Noise Floor Counts Planes, Not Sources

**Verdict:** The diagnosis is correct, and the current logic is too draconian for "Shadow" discovery.

* **The Logic Gap:** The "Multi-Plane" requirement (e.g., requiring Network + Endpoint) is excellent for determining *Management Status* (i.e., "Is this device fully under our control?"), but it is terrible for *Discovery* (i.e., "Is this thing talking on my network?").
* **The Risk:** If you have high-fidelity signals from three distinct network sources (Browser, Proxy, DNS) all agreeing that `asana.com` is active, rejecting it because they share a "Network" plane classification is a false negative.
* **Recommendation:**
* **Do not** simply lower the plane threshold to 1 (Fix Option B). That opens the floodgates to noise (e.g., a single errant DNS query creating an asset).
* **Implement Fix Option A (Count Distinct Sources):** This maintains the spirit of "triangulation" (requiring corroboration) without penalizing assets that only live on the network layer.
* *Refinement:* You might consider a weighted score. (e.g., Browser traffic is worth 1.0, DNS is worth 0.5. Admission requires > 1.0). But for now, **Count Distinct Sources** is the cleanest fix.



### Review of Issue 2: Key Normalization Breaks Reconciliation

**Verdict:** This is a classic "canonicalization vs. fidelity" conflict.

* **The Logic Gap:** The system is successfully being smart (normalizing `app.asana.com` → `asana.com`), but the Reconciliation engine is looking for the raw input, resulting in a mismatch.
* **The Risk:** If you just "update reconciliation expectations" (Fix Option B), you risk losing the granular detail of *what* was actually accessed. You want to know that users are hitting `app.asana.com`, not just the generic `asana.com`.
* **Recommendation:**
* **Implement Fix Option A (Emit Alias Metadata):** This is the robust architectural choice.
* The Asset should be created as `asana.com` (Canonical Key).
* It should attach `app.asana.com` as an `observed_alias` or within a `dns_names` list.
* This allows the Reconciliation engine to match against the *alias* if the primary key doesn't match, preventing the "Key Normalization Mismatch."
