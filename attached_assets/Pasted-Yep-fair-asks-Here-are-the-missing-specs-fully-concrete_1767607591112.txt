Not merged. **Recomputed** after merge using the same policy already documented: governed = idp OR cmdb; zombie = governed AND stale AND ongoing_finance; shadow = ungoverned AND recent. 

### Deterministic winner

Winner = asset with lexicographically smallest `asset.asset_id` (or if missing, smallest deterministic identifier in identifiers). This is stable across runs.

## 2) No-domain fallback (explicit behavior)

If `registered_domain` is missing / empty:

* **No merging.** Asset stays standalone.
* **No synthetic merge_key from name.** We do not create “name-based fan-in” because that reintroduces the exact instability that caused the prior failure.

What we *do* do:

* Keep `asset.name` as whatever admission selected (likely normalized name).
* Ensure `identifiers.hostnames` includes original_name if available (preservation, not transformation).

**Impact:** zombies that “lack domains” are still admitted (critical), but we do not risk accidental merges of unrelated apps with similar names.

This is consistent with “NO DOMAIN GUILLOTINE” and “entities without domains must NOT be rejected.”

## 3) Domain list ordering after merge (canonical at index 0)

We enforce a **single canonical domain** for the merged group and make it `domains[0]`.

### Canonical domain selection (for a merge group)

Pick the first available in this priority order:

1. Domain extracted from **IdP matched_records domain** (if valid)
2. Else domain extracted from **CMDB matched_records domain**
3. Else fallback to the winner asset’s current canonical (its `name` if it’s a domain, else its `identifiers.domains[0]`)
4. Else none

(Admission already has logic intended for this extraction. )

### Ordered union rule

After selecting `canonical`:

Final `identifiers.domains` list is:

1. `[canonical]` (if present)
2. then **winner’s existing domains in order**, excluding canonical
3. then all other domains from other assets, **stable-sorted** (lexicographic) excluding ones already included

So in your example:

* A: `[foo.com, foo.io]`
* B: `[foo.net, foo.com]`
* canonical: `foo.com`
* result: `[foo.com, foo.io, foo.net]`

**Impact:** deterministic ordering, “discovery-first” bias via winner precedence, and no churn.
