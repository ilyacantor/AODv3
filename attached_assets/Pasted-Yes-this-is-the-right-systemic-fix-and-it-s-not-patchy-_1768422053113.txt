## Critical refinements (do these, or you’ll chase new bugs)

### 1) Don’t merge by “primary registered domain” — merge by **overlapping registered domain sets**

Your `fastbox` example proves why:

* Asset A has `['fastbox.com','fastbox.org']`
* Asset B has `['fastbox.org']`

If you group by one “domain_key”, you can still miss merges when the chosen primary differs.
So the merge must be based on **connectivity**: any shared registered domain links them.

**Implementation detail:** union-find / connected components over registered domains.

---

### 2) Governance merge rule should be OR on authoritative `cmdb_present/idp_present`, not “authoritative wins”

“Authoritative wins over heuristic” is good intent, but the safe deterministic rule is:

* `cmdb_present = OR(cmdb_present)` across merged entities
* `idp_present = OR(idp_present)` across merged entities

Where `*_present` is **already gate-passed** (not “matched”).

This avoids accidentally dropping valid governance from one duplicate because the other had richer domains.

---

### 3) Preserve “matched-but-not-governing” separately

If you keep heuristic correlations for enrichment, don’t lose them in merge.

Add:

* `cmdb_matched = OR(cmdb_matched)`
* `cmdb_present = OR(cmdb_present)` (authoritative/governing only)

Same for IdP.

This prevents the earlier confusion of “match exists” vs “governed”.

### 4) Recompute classification after merge (you already said this — make it explicit)

This makes the bug class impossible to reintroduce.
