# AOD + Farm Operating Guide

This guide covers day-to-day operation of AOD (Asset Ownership Discovery) and Farm (test data generator).

---

## System Overview

| Component | URL | Purpose |
|-----------|-----|---------|
| **AOD** | `discover.autonomos.tech` | Discovery engine - processes snapshots, classifies assets |
| **Farm** | `farmv2.onrender.com` | Test data generator - creates realistic enterprise snapshots |

**Data Flow:**
```
Farm (Generate) → Snapshot → AOD (Process) → Catalog → Triage
```

---

## Part 1: Farm Operations

### What Farm Does

Farm generates synthetic but realistic enterprise data snapshots containing:
- **Discovery observations** (browser logs, DNS, network scans, proxy logs)
- **IdP records** (SSO/identity provider data)
- **CMDB records** (asset registry)
- **Finance transactions** (expense reports, invoices)
- **Cloud inventory** (AWS, Azure, GCP resources)

### Accessing Farm

1. Navigate to `farmv2.onrender.com`
2. Farm uses autoscale - first request may take 10-15 seconds to wake
3. AOD shows a cyan loading toast during Farm cold starts

### Generating Snapshots

| Parameter | Options | Description |
|-----------|---------|-------------|
| **Scale** | Small, Medium, Enterprise | Number of assets (50 → 500 → 1000+) |
| **Realism** | Clean, Messy, Adversarial | Data quality/conflict level |
| **Tenant** | Auto-generated | Unique tenant ID (e.g., `SysFlow-HGV9`) |

**Recommended for testing:**
- **Enterprise + Messy** - Realistic scale with data quality issues
- **Enterprise + Adversarial** - Stress test with intentional conflicts

### Snapshot Contents

Each snapshot includes ground truth expectations:
- Which assets should be admitted vs rejected
- Which assets should be classified as Shadow/Zombie/Governed
- Reason codes for each classification

### Farm API Endpoints

| Endpoint | Purpose |
|----------|---------|
| `GET /api/tenants` | List available tenants |
| `GET /api/snapshots?tenant_id=X` | List snapshots for tenant |
| `GET /api/snapshot/{id}` | Fetch full snapshot data |
| `GET /api/expectations/{snapshot_id}` | Get ground truth expectations |

## Part 2: AOD Operations

### Main Workflows

#### 1. Select Tenant
- Choose from dropdown (populated from Farm)
- Tenants are generated by Farm snapshots

#### 2. Run Discovery
- Click "Run Pipeline" to process selected snapshot
- Pipeline stages: Validate → Normalize → Index → Correlate → Admit → Classify → Output

#### 3. View Results
- **Lifecycle Cards**: Governed, Shadow, Zombie counts
- **Catalog**: Full list of admitted assets
- **Triage**: Actionable findings requiring review

### UI Navigation

| Tab | Purpose |
|-----|---------|
| **Dashboard** | Overview with lifecycle metrics |
| **Catalog** | Full asset inventory with search/filter |
| **Triage** | Prioritized findings for review |
| **Switchboard** | Policy configuration |
| **Reconcile** | Compare AOD results vs Farm expectations |

### Running Reconciliation

Reconciliation compares AOD's results against Farm's ground truth:

1. Select tenant and run pipeline
2. Navigate to Reconcile tab
3. Click "Run Reconciliation"
4. Review results:
   - **Matched**: AOD agrees with Farm
   - **Missed**: Farm expected, AOD didn't find
   - **FP (False Positive)**: AOD found, Farm didn't expect

### Reading Reconciliation Reports

| Metric | Target | Description |
|--------|--------|-------------|
| **Classification Accuracy** | >98% | Shadow/Zombie/Governed matches |
| **Admission Accuracy** | >99% | Cataloged vs Rejected matches |
| **Combined Accuracy** | >98.5% | Overall accuracy |

### Common Reconciliation Issues

| Issue | Cause | Resolution |
|-------|-------|------------|
| `KEY_NORMALIZATION_MISMATCH` | Domain aliasing differences | Check alias collapsing rules |
| `IDP_CORRELATION_MISMATCH` | IdP matching logic differs | Review correlation config |
| `FP_FROM_CLEAN` | Policy difference | Document as known difference |

## Part 3: Policy Switchboard

The Switchboard (`/switchboard`) controls all admission and classification logic.

### Key Policy Settings

| Category | Setting | Default | Description |
|----------|---------|---------|-------------|
| **Activity Windows** | `discovery_activity_window_days` | 90 | Days to consider for activity |
| | `zombie_window_days` | 90 | Inactivity threshold for zombie |
| **Finance Thresholds** | `minimum_spend` | $200 | Minimum monthly spend for admission |
| | `finance_gap_threshold` | $200 | Threshold to trigger finance finding |
| **Admission Gates** | `require_corroboration` | true | Require 2+ sources for admission |

### Alias Collapsing

Alias collapsing merges related domains to a canonical parent:

```
zoom.us → zoom.com
atlassian.net → atlassian.com  
office365.com → microsoft.com
```

Configure in `config/policy_master.json` under `alias_rules`.

## Part 4: Troubleshooting

### Farm Won't Respond

**Symptom:** Timeout or 502/503 errors
**Cause:** Farm is on autoscale, cold starting
**Resolution:** Wait 10-15 seconds, AOD auto-retries

### No Tenants in Dropdown

**Symptom:** Empty tenant selector
**Cause:** Farm hasn't generated snapshots yet
**Resolution:** Go to Farm, generate a snapshot first

### Reconciliation Shows Many Misses

**Symptom:** >2% missed assets
**Cause:** Policy drift between AOD and Farm
**Resolution:**
1. Check known policy differences (see below)
2. Review alias collapsing rules
3. Check admission gate settings

### Pipeline Fails

**Symptom:** Run stuck in "processing" or error state
**Cause:** Usually snapshot format issue or DB error
**Resolution:**
1. Check server logs
2. Verify snapshot format matches expected schema
3. Restart workflow if needed

## Part 5: Known Policy Differences with Farm

These are **documented decisions**, not bugs:

### Category A: Infrastructure Variants (Collapse to Parent)

| Domain | AOD Key | Rationale |
|--------|---------|-----------|
| `zoom.us` | `zoom.com` | TLD variant |
| `zoomapp.io` | `zoom.com` | App infrastructure |
| `zoom-meetings.net` | `zoom.com` | Meeting infrastructure |
| `adobelogin.com` | `adobe.com` | Auth portal |

**Farm should:** Accept parent domain as valid match

### Category B: Distinct Products (Standalone)

| Domain | AOD Behavior | Rationale |
|--------|--------------|-----------|
| `trello.com` | Standalone asset | Different attack surface than Atlassian |
| `bitbucket.org` | Standalone asset | Separate codebase/stack |

**Farm should:** Expect these as separate from `atlassian.com`

### Category C: Legacy Products (Collapse for Zombie Monitoring)

| Domain | AOD Key | Rationale |
|--------|---------|-----------|
| `hipchat.com` | `atlassian.com` | Discontinued product |

**Farm should:** Accept parent domain for zombie detection

## Part 6: Environment Configuration

### Required Environment Variables

| Variable | Purpose | Example |
|----------|---------|---------|
| `DATABASE_URL` | PostgreSQL connection | `postgresql://...` |
| `FARM_URL_PROD` | Production Farm URL | `https://farmv2.onrender.com` |
| `FARM_URL_DEV` | Development Farm URL | `http://localhost:8000` |
| `FARM_URL_MODE` | Which URL to use | `prod`, `dev`, `auto` |

### URL Mode Logic

| Mode | Behavior |
|------|----------|
| `prod` | Always use `FARM_URL_PROD` |
| `dev` | Always use `FARM_URL_DEV` |
| `auto` | Use prod if `REPLIT_DEPLOYMENT` is set, else dev |

## Part 7: Quick Reference

### Starting AOD

```bash
PYTHONPATH=/home/runner/workspace/src python -m uvicorn src.main:app --host 0.0.0.0 --port 5000 --reload
```

### API Health Check

```bash
curl http://localhost:5000/api/health
```

### List Tenants

```bash
curl http://localhost:5000/api/farm/tenants
```

### Trigger Pipeline Run

```bash
curl -X POST http://localhost:5000/api/runs/start \
  -H "Content-Type: application/json" \
  -d '{"tenant_id": "SysFlow-HGV9", "snapshot_id": "..."}'
```

### Run Reconciliation

```bash
curl http://localhost:5000/api/reconcile/{run_id}
```
