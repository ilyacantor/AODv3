▶ User Guide: Console

What This Tab Does

The Console is your command center for running asset discovery. Here you fetch data from various sources and process it through AOD's discovery engine to build a complete inventory of your organization's software assets.

How Discovery Works

AOD collects signals from multiple sources — identity providers (Okta, Azure AD), expense systems, browser telemetry, network logs, CMDBs, and cloud inventories. It then correlates these signals to identify unique applications, even when they're named differently across sources.

Key Concepts

  • Tenant: Your organization or business unit. Each tenant has its own isolated data.
  • Snapshot: A point-in-time collection of discovery data from Farm (test data) or production sources.
  • Discovery Run: A single execution of the discovery pipeline that processes a snapshot into cataloged assets.

Lifecycle Metrics

  • Ingested: Raw evidence collected from discovery sources (DNS, network, IdP, etc).
  • Validated: Observations that passed format checks and are ready for processing.
  • Rejected: Observations that failed validation — click to see why.
  • Cataloged: Final confirmed assets admitted to your catalog.

Classifications

  • Shadow: Assets in use but NOT governed — no SSO, not in CMDB. These are security blind spots.
  • Zombie: Assets that are governed but inactive for 90+ days — potential cost savings.
  • Security Risks: Identity gaps, shadow spending, and data conflicts requiring attention.
  • Governance: CMDB coverage gaps, ownership issues, and duplicate records.
Tip: Click any metric card to drill down and see the specific assets that make up that count.

▶ User Guide: Triage

What This Tab Does

The Triage console is where you make decisions about discovered assets. AOD surfaces issues that need human judgment before assets can be safely connected to your integration platform (AAM). Think of it as a security checkpoint.

The Three Workqueues

  • Blocking — Cannot Connect: Critical issues that prevent automatic connection. These assets have identity gaps (users accessing without SSO), unaccounted spending, or conflicting data from multiple sources. AAM will not connect these until resolved.
  • Judgment Required: Assets that could connect but need human review first. Includes zombies (inactive but still licensed), duplicates, and ambiguous classifications. You decide: keep, retire, or investigate further.
  • Informational — Non-Blocking: Minor issues that don't prevent connection but should be acknowledged. CMDB gaps, missing ownership, governance hygiene items. These can be batch-acknowledged.

Available Actions

  • Sanction: Approve a shadow IT asset for official use. This transitions it from "ungoverned" to "approved" status, allowing AAM to connect.
  • Ban: Explicitly block an asset. Users should stop using it, and AAM will never connect to it. Use for security risks or policy violations.
  • Deprovision: Retire a zombie asset. Confirms it's no longer needed and can be removed from licenses and access lists.
  • Assign: Route an item to a specific team member for investigation. Useful when you need more information before deciding.
  • Defer: Postpone a decision for a specified number of days. The item will reappear after the deferral period.
  • Acknowledge: Mark informational items as seen. Confirms you're aware of the issue without taking action.

How Triage Affects AAM

Every asset AOD discovers is sent to AAM (Adaptive API Mesh) as a "Connection Candidate." However, assets with blocking findings are flagged as inventory_only — AAM receives them for visibility but will not attempt to connect. Once you resolve the blocking issue (Sanction, Ban, or Override), the asset becomes provision-ready.

Important: Triage actions are run-scoped. Each discovery run creates a fresh set of assets. Actions taken on one run do not automatically carry forward to future runs.
Tip: Use the batch "Acknowledge Selected" feature in the Informational section to quickly clear low-priority items.

▶ User Guide: Policy

What This Tab Does

The Policy Switchboard controls how AOD classifies assets and generates findings. These rules determine what gets flagged as Shadow IT, when an asset is considered a Zombie, and what issues trigger blocking vs. informational alerts.

Policy Categories

  • Governance Rules: Define the "Governance Trinity" — what qualifies an asset as governed (CMDB presence, IdP/SSO validation, vendor lifecycle control).
  • Activity Thresholds: Configure the inactivity window for zombie detection (default: 90 days). Assets governed but inactive beyond this threshold are flagged as zombies.
  • Finance Rules: Set thresholds for finance tracking. Assets with spending above the threshold but no governance are flagged as shadow IT spend risks.
  • Finding Severity: Control which findings are blocking (Red) vs. review (Yellow) vs. informational (Green). This directly affects AAM execution signals.

IdP Governance Policy

A critical setting that controls how SSO matches determine governance:

  • Strict Mode: Assets must have CMDB presence AND IdP match to be governed. Catches more shadow IT but may flag legitimate apps.
  • Loose Mode: IdP match alone is sufficient for governance. Reduces false positives but may miss shadow IT that's been added to SSO without proper approval.

Custom Exclusions

Specify domains or patterns that should be excluded from discovery entirely. Useful for internal infrastructure, development environments, or known-good systems that don't need governance tracking.

Policy Manifest

All policy settings are compiled into a versioned manifest that's exported to AAM. This ensures AAM applies the same governance rules when making connection decisions. Changes to policy take effect on the next discovery run.

Caution: Changing policy settings affects future discovery runs, not historical data. If you change the zombie threshold from 90 to 60 days, you'll need to re-run discovery to see updated classifications.
▶ User Guide: Handoff

What This Tab Does

The Handoff tab is where AOD transfers its discovery results to AAM (Adaptive API Mesh). Every discovered asset becomes a "Connection Candidate" that AAM will use to establish integrations. This is the bridge between knowing what exists and actually connecting to it.

Understanding Connection Candidates

Each candidate represents a single discovered asset with all the information AAM needs to connect:

  • Asset Key: The canonical identifier (usually the primary domain)
  • Governance Status: Governed, Shadow, Zombie, or Edge
  • Execution Signal: Whether AAM should auto-connect or wait for human approval
  • Findings: Any issues discovered during triage
  • SOR Tagging: Whether this asset is a System of Record for a data domain

Fabric Planes

AOD identifies "Fabric Planes" — integration motherships that aggregate many other applications. Instead of connecting to 500 individual apps, AAM can connect to 3-4 fabric planes that already have access to everything:

  • iPaaS: Integration platforms like MuleSoft, Workato, Tray.io
  • API Gateway: API management platforms like Kong, Apigee, AWS API Gateway
  • Data Warehouse: Analytics platforms like Snowflake, BigQuery, Databricks
  • Event Bus: Streaming platforms like Kafka, AWS EventBridge, Azure Event Hubs

Execution Signals

  • provision (green): No blocking issues. AAM can auto-connect this asset.
  • inventory_only (amber): Blocking findings exist. AAM receives this for visibility but will not connect until triage is resolved.

Exporting to AAM

Click "Export to AAM" to send all connection candidates for the selected run. AAM receives the complete inventory — both provision-ready and inventory-only candidates. AAM uses the execution signals to determine which assets to connect automatically vs. which to hold for human review.

Tip: Use the status filter to see only candidates that need attention before export.
